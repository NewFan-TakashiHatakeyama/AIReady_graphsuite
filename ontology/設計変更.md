## 1. 変更の背景と動機

### 1.1 設計課題の発見

現行設計では、同一ドキュメントに対して **Governance と Ontology が独立にテキスト解析を実行** しており、以下の無駄が発生している。

```
現行設計（同一ドキュメントに対して 2 回処理）:

DynamoDB Streams (FileMetadata INSERT/MODIFY)
    │
    ├──▶ Governance: detectSensitivity
    │     ├── S3 からファイル取得             ← ★ 1 回目
    │     ├── テキスト抽出
    │     ├── GiNZA (ja-ginza) + Presidio    ← ★ NLP モデルロード 1 回目
    │     ├── PII 検知 → SensitivityScore
    │     └── テキスト破棄                   ← ★ 解析結果を蓄積しない
    │
    └──▶ Ontology: nounExtractor
          ├── S3 からファイル取得             ← ★ 2 回目（同じファイル）
          ├── テキスト抽出
          ├── GiNZA (ja_ginza_electra) + spaCy ← ★ NLP モデルロード 2 回目
          ├── NER + 名詞チャンク → エンティティ候補
          └── テキスト破棄                   ← ★ ここでもテキストを捨てている
```

| 問題 | 詳細 |
|------|------|
| **S3 二重取得** | 同一ドキュメントの Raw Payload を 2 つの Lambda がそれぞれ取得 |
| **NLP モデル二重ロード** | Governance と Ontology で別々に GiNZA をロード（標準 vs Transformer） |
| **テキスト抽出の重複** | xlsx, docx, pdf 等のテキスト抽出を 2 回実行 |
| **PII と NER の分断** | 同じ「田中太郎」を PII と NER で別々に検出し、後続で突合が必要 |
| **テキスト資産の未活用** | 取得したテキストから要約やベクトル表現を生成する機会を逸失 |

### 1.2 変更の方針

> **1 ドキュメント = 1 回の解析 = 1 つの解析結果セット**

ドキュメントのコンテンツ解析を **Governance の `detectSensitivity` に一元化** し、Ontology はその解析結果を消費して **エンティティ解決・ゴールドマスタ管理に特化** する。
さらに、解析時に **ドキュメント要約** と **Embedding（ベクトル表現）** を生成し、v2.0 のナレッジグラフ構築およびベクトル検索の基盤を v1.0 段階から整備する。

## 2. 変更方針の要約

### 2.1 責務の再定義

```
変更前:
  Governance = PII/機微検知（テキスト解析 → リスクスコア）
  Ontology   = NER + 名詞抽出（テキスト解析 → エンティティ候補）
               + エンティティ解決 + ゴールドマスタ管理

変更後:
  Governance = ドキュメントコンテンツ解析の一元担当
               （PII 検知 + NER + 名詞チャンク + 要約 + Embedding 生成）
               → 解析結果を DocumentAnalysis テーブル / S3 Vectors に蓄積

  Ontology   = エンティティ管理・知識構造化に特化
               （エンティティ解決 + ゴールドマスタ + ナレッジグラフ）
               → 蓄積済みの解析結果を消費するのみ
```

### 2.2 変更サマリ

| 変更項目 | Governance 側 | Ontology 側 |
|---------|-------------|------------|
| **detectSensitivity** | NER + 名詞チャンク + 要約 + Embedding を追加 | — |
| **nounExtractor** | — | **廃止**（解析を Governance に移管） |
| **中間ストレージ** | `AIReady-DocumentAnalysis` テーブルに解析結果を保存 | 同テーブルから解析結果を読み取り |
| **S3 Vectors** | Embedding を格納 | v2.0 でベクトル検索に活用 |
| **entityResolver** | — | 入力ソースが SQS 直接 → DocumentAnalysis 経由に変更 |
| **NLP 依存** | GiNZA + spaCy + Presidio + Bedrock を集約 | **NLP 依存ゼロ** |

## 3. 現状の問題分析

### 3.1 リソースの二重消費

| リソース | Governance (detectSensitivity) | Ontology (nounExtractor) | 統合後 |
|---------|------------------------------|--------------------------|--------|
| S3 GetObject | 1 回/doc | 1 回/doc | **1 回/doc** |
| テキスト抽出 | 1 回/doc | 1 回/doc | **1 回/doc** |
| GiNZA ロード | ja-ginza (標準) | ja_ginza_electra (Transformer) | **1 回のみ (Transformer に統一)** |
| Lambda メモリ | 3072 MB | 1024 MB | **3072〜4096 MB (1 Lambda)** |
| Docker イメージ | あり (Presidio + GiNZA) | あり (GiNZA + spaCy) | **1 イメージに統合** |

### 3.2 PII と NER の分断による複雑性

```
現状の問題:

  Governance detectSensitivity:
    「田中太郎」 → PII (person_name) 検知 → SQS → entityResolver

  Ontology nounExtractor:
    「田中太郎」 → NER (Person) 検知 → SQS → entityResolver

  entityResolver:
    2 つの経路から同一人物の候補が到着
    → canonical_hash で突合して同一エンティティと判定する必要がある
    → 到着タイミングのズレ、順序保証の複雑性

統合後:
  Governance detectSensitivity:
    「田中太郎」
    → PII (person_name) + NER (Person) を同一コンテキストで同時検出
    → pii_flag = true のエンティティ候補として 1 件だけ出力
    → entityResolver に 1 回だけ到着
```

### 5.3 GiNZA モデルの統一

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| Governance | `ja-ginza`（標準モデル） | **`ja_ginza_electra`**（Transformer モデル） |
| Ontology | `ja_ginza_electra` | **モデル不要**（NLP 依存なし） |

Transformer モデルへの統一により PII 検知の精度も向上する。Docker イメージサイズの増加（約 +500MB）はあるが、2 つの Lambda で別々にモデルを保持するよりも総リソース消費は削減される。

### 5.4 PII と NER の統合検出ロジック

同一テキスト上で PII 検知と NER を同時実行し、結果を統合する。

```python
# 統合検出の概念コード
def analyze_document(text: str, language: str) -> AnalysisResult:
    nlp = _get_nlp(language)  # GiNZA or spaCy
    doc = nlp(text)

    # NER エンティティ抽出
    ner_entities = extract_ner_entities(doc)

    # Presidio PII 検知
    pii_findings = detect_pii(text, language)

    # 名詞チャンク抽出
    noun_chunks = extract_noun_chunks(doc)

    # PII と NER の統合: 同一スパンの重複を排除
    merged_candidates = merge_pii_and_ner(
        ner_entities=ner_entities,
        pii_findings=pii_findings,
        noun_chunks=noun_chunks,
    )
    # PII として検知された候補には pii_flag = True を付与
    # NER のみで検出された候補は pii_flag = False

    return AnalysisResult(
        candidates=merged_candidates,
        pii_summary=pii_findings,
        sensitivity_score=calculate_sensitivity(pii_findings),
    )
```

**統合ルール:**

| ケース | PII 検知 | NER 検知 | 結果 |
|--------|---------|---------|------|
| PII のみ | `person_name: "田中太郎"` | — | `pii_flag=true, extraction_source="governance"` |
| NER のみ | — | `Person: "田中太郎"` | `pii_flag=true, extraction_source="ner"` (Person は PII 候補) |
| 両方検知 | `person_name: "田中太郎"` | `Person: "田中太郎"` | **1 件に統合**: `pii_flag=true, extraction_source="governance+ner"` |
| ビジネス名詞のみ | — | `Product: "AI Ready"` | `pii_flag=false, extraction_source="ner"` |

## 6. Ontology 側の変更（nounExtractor 廃止・パイプライン再構成）

### 6.1 Lambda 構成の変更

```
変更前（5 Lambda）:
  ├── schemaTransform      (DynamoDB Streams)
  ├── lineageRecorder      (同期呼び出し)
  ├── nounExtractor        (SQS, 1024MB, GiNZA+spaCy)  ← ★ 廃止
  ├── entityResolver       (SQS, Aurora 接続)
  └── batchReconciler      (EventBridge 日次)

変更後（4 Lambda）:
  ├── schemaTransform      (DynamoDB Streams)        ← 変更なし
  ├── lineageRecorder      (同期呼び出し)            ← 変更なし
  ├── entityResolver       (SQS, Aurora 接続)        ← 入力ソースの変更
  └── batchReconciler      (EventBridge 日次)        ← 変更なし
```

### 6.2 廃止対象

| 対象 | 種別 | 理由 |
|------|------|------|
| `AIReadyOntology-nounExtractor` Lambda | Lambda 関数 | 解析処理を Governance に移管 |
| `AIReadyOntology-NounExtractionQueue` | SQS Standard | nounExtractor のトリガーキューが不要に |
| `AIReadyOntology-NounExtraction-DLQ` | SQS Standard (DLQ) | 上記キューの DLQ が不要に |
| `nlp-layer` Lambda Layer | Lambda Layer | GiNZA / spaCy モデルが不要に |
| `src/handlers/noun_extractor.py` | ソースコード | Lambda ハンドラ廃止 |
| `src/services/ner_pipeline.py` | ソースコード | NER パイプラインを Governance に移管 |
| `src/services/domain_dictionary.py` | ソースコード | ドメイン辞書管理を Governance に移管 |

### 6.3 schemaTransform の変更

nounExtractor 廃止に伴い、schemaTransform から NounExtractionQueue へのエンキュー処理を削除する。

```
変更前:
  schemaTransform
    ├── 共通スキーマ変換 → UnifiedMetadata 保存
    ├── lineageRecorder 呼び出し
    └── SQS 送信 (NounExtractionQueue)        ← ★ 削除

変更後:
  schemaTransform
    ├── 共通スキーマ変換 → UnifiedMetadata 保存
    └── lineageRecorder 呼び出し
```

### 6.4 entityResolver の入力変更

| 項目 | 変更前 | 変更後 |
|------|--------|--------|
| 入力キュー | `AIReadyOntology-EntityResolutionQueue.fifo` | **同じキュー名（変更なし）** |
| メッセージ送信元 | Ontology nounExtractor + Governance PII | **Governance detectSensitivity のみ** |
| メッセージ形式 | 2 種類（NER 候補 / PII 候補） | **1 種類に統合**（PII フラグ付きエンティティ候補） |

統合後のメッセージ仕様:

```python
{
    "message_type": "entity_resolution_request",
    "source": "document_analysis",
    "tenant_id": "tenant-abc",
    "candidate_id": "uuid-v4",
    "source_item_id": "item456",
    "surface_form": "田中太郎",
    "normalized_form": "タナカタロウ",
    "entity_type": "person",
    "pii_flag": True,
    "pii_category": "person_name",        # PII の場合のみ
    "extraction_source": "governance+ner", # 統合検出の場合
    "confidence": 0.95,
    "mention_count": 3,
    "context_snippet": "...担当者の田中太郎さんが...",
    "language": "ja",
    "analysis_id": "uuid-of-analysis",    # DocumentAnalysis レコード参照
    "lineage_id": "uuid-of-lineage",
    "requested_at": "2026-02-23T10:00:00Z"
}
```

### 6.5 UnifiedMetadata の拡張フィールド

DocumentAnalysis の結果を参照し、UnifiedMetadata に以下のフィールドを追加する。

| フィールド | 型 | 説明 | 設定タイミング |
|-----------|-----|------|-------------|
| `document_summary` | String | ドキュメント要約テキスト（最大 500 文字） | schemaTransform（DocumentAnalysis 参照）or batchReconciler |
| `summary_language` | String | 要約の言語（ja / en） | 同上 |
| `topic_keywords` | List\<String\> | トピックキーワード（上位 10 件） | 同上 |
| `embedding_ref` | String | S3 Vectors のキー参照 | 同上 |
| `analysis_id` | String | DocumentAnalysis レコードの参照 ID | 同上 |
| `summary_generated_at` | String | 要約生成日時 (ISO 8601) | 同上 |

> **タイミングの考慮**: Governance の detectSensitivity と Ontology の schemaTransform は同じ DynamoDB Streams をトリガーとするため、schemaTransform 実行時に DocumentAnalysis がまだ存在しない可能性がある。その場合は `document_summary` 等を空のまま保存し、**batchReconciler の日次バッチで DocumentAnalysis を参照して補完** する。

## 7. 中間ストレージ設計（DocumentAnalysis）

### 7.1 テーブル設計

Governance が解析結果を保存し、Ontology が消費するための共有 DynamoDB テーブル。

| 項目 | 値 |

## 8. ドキュメント要約・Embedding 基盤

### 8.1 要約生成

| 項目 | 仕様 |
|------|------|
| モデル | Amazon Bedrock — Claude 3 Haiku |
| 最大入力 | 100,000 文字（超過分は末尾切り捨て） |
| 要約長 | 最大 500 文字 |
| 言語 | 入力テキストと同一言語で要約 |
| トピックキーワード | NER + 名詞チャンクの出現頻度上位 10 件 |
| コスト | 約 $0.0005 / ドキュメント (input: ~2K tokens, output: ~200 tokens) |

**要約プロンプト設計:**

```
以下のドキュメントを {language} で 500 文字以内に要約してください。
要約には以下を含めてください:
- ドキュメントの主題・目的
- 主要な関係者・組織（固有名詞）
- 重要な数値・日付・決定事項

ドキュメント:
{text}
```

### 8.2 Embedding 生成

| 項目 | 仕様 |
|------|------|
| モデル | Amazon Bedrock — Titan Embeddings V2 |
| 入力 | ドキュメント要約テキスト（全文ではなく要約を使用） |
| 次元数 | 1024 |
| 格納先 | S3 Vectors |
| コスト | 約 $0.00002 / ドキュメント |

> **設計判断**: Embedding の入力に全文ではなく**要約テキスト**を使用する理由:
> - 要約はノイズが除去されており、トピック類似度の判定精度が高い
> - トークン数が少なく、Embedding 生成コストが低い
> - 全文 Embedding は長文の中間部分の情報が埋没する問題（Lost in the Middle）を回避
